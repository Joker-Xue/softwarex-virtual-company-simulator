## 1. 红线问题（必改）

### 🔴 R0-1 论文 Table 3 与代码 `_get_mbti_weights` 不一致（自查发现）

**事实**：
- 论文 Table 3 在 Δ 矩阵里给了 S/N 维度权重：`S → work +4`、`N → meeting +6`，并据此给出示例 `ISTJ work = 30+15(I)+4(S)+10(T)+10(J) = 69`，对应 work 概率 49.6%
- 实际代码 [src/backend/app/engine/agent_ai.py:90-111](../src/backend/app/engine/agent_ai.py) **完全没有处理 mbti[1] (S/N)**：
  ```python
  weights = {"work": 30, "chat": 20, "rest": 15, "move_to": 25, "meeting": 10}
  if mbti[0] == "E": ...   # 处理
  # ❌ 没有 if mbti[1] == "S"/"N" 分支
  if mbti[2] == "T": ...   # 处理
  if mbti[3] == "J": ...   # 处理
  ```
- 代码侧 S/N 只出现在 `agent_ai.py:581` 与 `simulation_loop.py:55,79`，作用是**任务效率乘数 η_work（公式 3）和经验值乘数 η_xp（公式 5）**，而不是 Δ 矩阵
- 实测产物 [results/behavior_consistency.csv:10](../results/behavior_consistency.csv) ISTJ expected_work_pct = **48.15%**（65/135），与论文 49.6% 不一致

**严重度推断**（非已核实事实）：审稿人没指出，但任何审稿人去跑代码就会发现 Table 3 含 S/N 而代码不含。SoftwareX 模板没有"代码与论文一致"的明文条款，但属于学术论文通行底线。

**解法**（二选一，推荐 A）：
- **A. 代码补齐 S/N**：在 `_get_mbti_weights` 加：
  ```python
  if mbti[1] == "S":
      weights["work"] += 4
  else:
      weights["meeting"] += 6
  ```
  好处：保留 Table 3 全貌；让 16 类 MBTI 在动作分布上完全可分。代价：重跑 4 个实验。
- **B. 论文删 S/N 行**：Table 3 移除 S/N 两行，示例改为 `ISTJ work = 30+15(I)+10(T)+10(J) = 65 → 48.1%`。代价：Δ 矩阵只用 3 维，相关工作部分关于"explicit MBTI parameterization"的力度变弱，且 reviewer 1 #4 的"论证 Δ 数值"反而更难答。

**附带工作**：选 A 后必须重跑 `behavior_consistency` / `ablation`，更新 Section 3 Example 1 的数字、Section 4.4、Table 5、Fig 3，以及 README/results 基线。
